## 1. Fondation stratégique

### 1.1 Ce que NEXUS résout

Les agents individuels sont puissants. Mais sans coordination, ils produisent :
- Des décisions d'architecture contradictoires
- Des efforts dupliqués entre divisions
- Des trous de qualité aux points de passation
- Aucun contexte partagé ni mémoire institutionnelle

**NEXUS élimine ces modes de défaillance** en définissant :
- **Qui** s'active à chaque phase
- **Quoi** ils produisent et pour qui
- **Quand** ils passent le relais et à qui
- **Comment** la qualité est vérifiée avant d'avancer
- **Pourquoi** chaque agent existe dans le pipeline (pas de passagers)

### 1.2 Principes fondateurs

| Principe | Description |
|-----------|-------------|
| **Intégrité du pipeline** | Aucune phase n'avance sans franchir sa porte qualité |
| **Continuité du contexte** | Chaque passation porte le contexte complet — aucun agent ne démarre à froid |
| **Exécution parallèle** | Les flux de travail indépendants tournent en parallèle pour compresser les délais |
| **La preuve plutôt que les affirmations** | Toute évaluation de la qualité exige des preuves, pas des assertions |
| **Échouer vite, corriger vite** | Maximum 3 tentatives par tâche avant escalade |
| **Source unique de vérité** | Une spec canonique, une liste de tâches, un document d'architecture |

### 1.3 L'effectif d'agents par division

| Division | Agents | Rôle NEXUS principal |
|----------|--------|--------------------|
| **Engineering** | Frontend Developer, Backend Architect, Mobile App Builder, AI Engineer, DevOps Automator, Rapid Prototyper, Senior Developer | Construire, déployer et maintenir tous les systèmes techniques |
| **Design** | UI Designer, UX Researcher, UX Architect, Brand Guardian, Visual Storyteller, Whimsy Injector, Image Prompt Engineer | Définir l'identité visuelle, l'expérience utilisateur et la cohérence de marque |

| **Project Management** | Studio Producer, Project Shepherd, Studio Operations, Experiment Tracker, Senior Project Manager | Orchestrer les calendriers, les ressources et la coordination transverse |
| **Testing** | Evidence Collector, Reality Checker, Test Results Analyzer, Performance Benchmarker, API Tester, Tool Evaluator, Workflow Optimizer | Vérifier la qualité par une évaluation fondée sur les preuves |
| **Support** | Support Responder, Analytics Reporter, Finance Tracker, Infrastructure Maintainer, Legal Compliance Checker, Executive Summary Generator | Pérenniser les opérations, la conformité et l'intelligence économique |

## 4. Phase 1 — Stratégie & Architecture

> **Objectif** : définir ce que l'on construit, comment c'est structuré et à quoi ressemble le succès — avant d'écrire la moindre ligne de code.

### 4.1 Agents actifs

| Agent | Rôle dans la phase | Sortie principale |
|-------|--------------|----------------|
| **Studio Producer** | Alignement du portefeuille stratégique | Plan de portefeuille stratégique |
| **Senior Project Manager** | Conversion spec-vers-tâches | Liste de tâches exhaustive |
| **Sprint Prioritizer** | Priorisation des fonctionnalités | Backlog priorisé (scoré RICE) |
| **UX Architect** | Architecture technique + fondation UX | Spec d'architecture + CSS Design System |
| **Brand Guardian** | Système d'identité de marque | Document de fondation de marque |
| **Backend Architect** | Architecture système | Spécification d'architecture système |
| **AI Engineer** | Architecture IA/ML (le cas échéant) | ML System Design |
| **Finance Tracker** | Planification du budget et des ressources | Plan financier avec projections de ROI |

### 4.2 Séquence d'exécution

```
STEP 1: Strategic Framing (Parallel)
├── Studio Producer → Strategic Portfolio Plan (vision, objectives, ROI targets)
├── Brand Guardian → Brand Foundation (purpose, values, visual identity system)
└── Finance Tracker → Budget Framework (resource allocation, cost projections)

STEP 2: Technical Architecture (Parallel, after Step 1)
├── UX Architect → CSS Design System + Layout Framework + UX Structure
├── Backend Architect → System Architecture (services, databases, APIs)
├── AI Engineer → ML Architecture (models, pipelines, inference strategy)
└── Senior Project Manager → Task List (spec → tasks, exact requirements)

STEP 3: Prioritization (Sequential, after Step 2)
└── Sprint Prioritizer → RICE-scored backlog with sprint assignments
    ├── Input: Task List + Architecture Spec + Budget Framework
    ├── Output: Prioritized sprint plan with dependency map
    └── Validation: Studio Producer confirms strategic alignment
```

### 4.3 Porte qualité de la phase 1

**Gardien de porte** : Studio Producer + Reality Checker (double validation)

| Critère | Seuil | Preuve requise |
|-----------|-----------|-------------------|
| L'architecture couvre toutes les exigences | 100 % de couverture de la spec | Liste de tâches du Senior PM recoupée |
| Système de marque complet | Logo, couleurs, typographie, voix définis | Livrable du Brand Guardian |

## 7. Phase 4 — Qualité & Durcissement

> **Objectif** : l'épreuve qualité finale. Le Reality Checker se positionne par défaut sur « NEEDS WORK » — vous devez prouver l'aptitude à la production avec des preuves écrasantes.

### 7.1 Agents actifs

| Agent | Rôle dans la phase | Sortie principale |
|-------|--------------|----------------|
| **Reality Checker** | Test d'intégration final (par défaut : NEEDS WORK) | Rapport d'intégration fondé sur la réalité |
| **Evidence Collector** | Preuves visuelles exhaustives | Package de preuves par captures d'écran |
| **Performance Benchmarker** | Test de charge + optimisation | Certification de performance |
| **API Tester** | Suite de régression API complète | Rapport de tests API |
| **Test Results Analyzer** | Agrégation des métriques qualité | Tableau de bord des métriques qualité |
| **Legal Compliance Checker** | Audit de conformité final | Certification de conformité |
| **Infrastructure Maintainer** | Contrôle d'aptitude à la production | Rapport d'aptitude de l'infrastructure |
| **Workflow Optimizer** | Revue de l'efficacité des processus | Recommandations d'optimisation |

### 7.2 La séquence de durcissement

```
STEP 1: Evidence Collection (Parallel)
├── Evidence Collector → Full screenshot suite (desktop, tablet, mobile)
├── API Tester → Complete endpoint regression
├── Performance Benchmarker → Load test at 10x expected traffic
└── Legal Compliance Checker → Final regulatory audit

STEP 2: Analysis (Parallel, after Step 1)
├── Test Results Analyzer → Aggregate all test data into quality dashboard
├── Workflow Optimizer → Identify remaining process inefficiencies
└── Infrastructure Maintainer → Production environment validation

STEP 3: Final Judgment (Sequential, after Step 2)
└── Reality Checker → Integration Report
    ├── Cross-validates ALL previous QA findings
    ├── Tests complete user journeys with screenshot evidence
    ├── Verifies specification compliance point-by-point
    ├── Default verdict: NEEDS WORK
    └── READY only with overwhelming evidence across all criteria
```

### 7.3 Porte qualité de la phase 4 (LA PORTE FINALE)

**Gardien de porte** : Reality Checker (autorité unique)

| Critère | Seuil | Preuve requise |
|-----------|-----------|-------------------|
| Parcours utilisateurs complets | Tous les chemins critiques fonctionnent | Captures d'écran de bout en bout |
| Cohérence cross-device | Desktop + tablette + mobile | Captures d'écran responsive |
| Performance certifiée | P95 < 200ms, uptime > 99,9 % | Résultats du test de charge |
| Sécurité validée | Zéro vulnérabilité critique | Rapport de scan de sécurité |
| Conformité certifiée | Toutes les exigences réglementaires remplies | Rapport du Legal Compliance Checker |
| Conformité à la spécification | 100 % des exigences de la spec | Vérification point par point |

**Options de verdict** :
- **READY** — passer au lancement (rare au premier passage)
- **NEEDS WORK** — retour à la phase 3 avec une liste de correctifs précise (attendu)
- **NOT READY** — problèmes d'architecture majeurs, retour à la phase 1/2

**Attendu** : les premières implémentations nécessitent généralement 2 à 3 cycles de révision. Une note B/B+ est normale et saine.

### 12.2 Gestion des échecs de porte

```
IF gate FAILS:
  ├── Gate Keeper produces specific failure report
  ├── Agents Orchestrator routes failures to responsible agents
  ├── Failed items enter Dev↔QA loop (Phase 3 mechanics)
  ├── Maximum 3 gate re-attempts before escalation to Studio Producer
  └── Studio Producer decides: fix, descope, or accept with risk
```

| **NEXUS** | Network of EXperts, Unified in Strategy |
| **Porte qualité (Quality Gate)** | Point de contrôle obligatoire entre phases nécessitant une approbation fondée sur les preuves |
| **Boucle Dev↔QA** | Cycle continu développement-test où chaque tâche doit passer la QA avant d'avancer |
| **Passation (Handoff)** | Transfert structuré de travail et de contexte entre agents |
| **Gardien de porte (Gate Keeper)** | Agent(s) ayant autorité pour approuver ou rejeter l'avancement d'une phase |
| **Escalade** | Acheminement d'une tâche bloquée vers une autorité supérieure après épuisement des tentatives |
| **NEXUS-Full** | Activation complète du pipeline avec tous les agents |
| **NEXUS-Sprint** | Pipeline focalisé avec 15-25 agents pour le travail sur fonctionnalité/MVP |
| **NEXUS-Micro** | Activation ciblée de 5-10 agents pour des tâches spécifiques |
| **Intégrité du pipeline** | Principe selon lequel aucune phase n'avance sans franchir sa porte qualité |
| **Continuité du contexte** | Principe selon lequel chaque passation porte le contexte complet |
| **La preuve plutôt que les affirmations** | Principe selon lequel les évaluations de qualité exigent des preuves, pas des assertions |
